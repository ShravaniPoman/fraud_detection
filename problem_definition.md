# Fraud Detection: Problem Definition
**Project:** AI-Generated & Manipulated Medical Invoice Detection  
**Version:** 1.0  
**Last Updated:** 2026-03-10

---

## 1. Task

> Given a medical invoice or bill document (PDF/image), predict whether it is **authentic**, **manually tampered**, or **AI-generated/heavily rewritten**.

This is a document-level classification problem. The model inspects the content, layout, and structure of a single medical invoice and assigns it to one of three classes.

---

## 2. Input

| Field | Description |
|-------|-------------|
| **Primary input** | A single medical invoice or hospital bill as a scanned PDF or image (JPG/PNG) |
| **Optional secondary input** | A paired companion document (e.g., Explanation of Benefits / EOB) for cross-document consistency checking |
| **Derived inputs** | OCR-extracted text, bounding boxes/layout, and structured numeric fields parsed from the document |

**What the model is NOT given at inference time:**
- Access to claims history or provider databases
- Ground-truth JSON fields (these are only available during training)

---

## 3. Output / Target Labels

### Primary Label (class)

| Class | Name | Description |
|-------|------|-------------|
| `0` | Authentic | Original, untampered document with no edits |
| `1` | Manually Tampered | Human-edited document — amounts, dates, line items, or identity fields changed |
| `2` | AI-Generated / Rewritten | Document substantially generated or rewritten by a generative AI model |

### Secondary Label (fraud_type)
Only populated when `class != 0`. Used for fine-grained analysis and targeted evaluation.

| fraud_type | Description |
|------------|-------------|
| `amount_inflation` | One or more line-item prices or totals increased |
| `date_extension` | Admission/discharge/invoice dates altered |
| `lineitem_added` | A new (fictitious) procedure or charge inserted |
| `lineitem_removed` | A discount, adjustment, or line item deleted |
| `identity_tweak` | Patient name, ID, or policy number modified |
| `ai_full_doc` | Entire document generated by an LLM/AI tool |
| `ai_text_rewrite` | Human template filled with AI-generated text content |

---

## 4. Features

The model may use any combination of the following signals:

| Feature Type | Examples |
|--------------|---------|
| **Raw visual** | Pixel-level image of the document (resized to fixed dimensions) |
| **OCR text** | Full extracted text from the invoice |
| **Layout / bounding boxes** | Token positions and structure from OCR |
| **Numeric consistency** | `total_amount` vs. `sum(line_item totals)`, tax rate plausibility |
| **Date logic** | `invoice_date` vs. `discharge_date`, coverage window checks |
| **Metadata** | PDF creation tool, edit history, embedded fonts |
| **Cross-document delta** | Field mismatches between invoice and paired EOB (optional) |

---

## 5. Evaluation

### Primary Metric
**Macro F1-score** on the held-out **test set**, reported for:
- Each class individually (F1 for class 0, 1, 2)
- Binary fraud detection: F1 for `class in {1, 2}` vs. `class == 0`

### Secondary Metrics
- Precision and Recall per class
- Confusion matrix (3×3)
- Per `fraud_type` breakdown on class-1 examples

### Success Threshold (target for 5-week project)
- Binary fraud F1 ≥ 0.75 on the test set
- Model detects at least 2 distinct fraud types reliably

### What NOT to optimize
- Do not tune hyperparameters on the test set — use validation set only
- Test set is locked; run final evaluation only once

---

## 6. Data Split Strategy

| Split | Size | Purpose |
|-------|------|---------|
| Train | 70% | Model training |
| Validation | 15% | Hyperparameter tuning, model selection |
| Test | 15% | Final evaluation only |

**Key rule:** All variants of a given base invoice (clean + all fraud versions) must go into the **same split** to prevent leakage.

---

## 7. Out of Scope (for this project)

- Network-level fraud detection (duplicate submissions across insurers)
- Real patient data or PHI of any kind
- Provider-side coding fraud (upcoding, unbundling) from tabular claims
- Production deployment or real-time inference

---

## 8. Novelty vs. Existing Tools

| Existing tools | This project |
|----------------|-------------|
| Rule-based anomaly engines (duplicate invoices, vendor mismatch) | ML model on document content + layout |
| Generic invoice fraud detectors | Focused specifically on **medical** invoices |
| Proprietary black-box AI detectors (TruthScan, Klippa) | Open, labeled synthetic dataset with explicit fraud types |
| Single-document classifiers | Optional cross-document consistency (invoice vs. EOB) |
| No adversarial training | Adversarial loop: generate harder fakes → retrain |
